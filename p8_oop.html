PREREQ P8.1 · CREATING AND USING A CLASS

P8.1 Creating and using a class

Everything in Python is an object, so you have been using objects all along — int, str, list, and dict are all classes that other people wrote. Object-oriented programming (OOP) is the paradigm behind them: group variables (data, called attributes) and functions (called methods) into a new custom data type called a class, then create objects (instances) from it. Making an object from a class is called instantiation. When you write a class you define the general behavior of a whole category of things; each object you create gets that behavior automatically, plus whatever unique traits you give it.

📌 THE FOUR WORDS THIS COURSE NEEDS

Class = the blueprint. Object (instance) = one thing built from the blueprint. Attribute = a variable attached to an object (my_dog.name — no parentheses). Method = a function attached to an object (my_dog.sit() — parentheses). Modules 1–5 never ask you to write classes, but every line like arr.mean() or df.shape reads instantly once these four words are solid.

Let's model a dog — not one dog in particular, but any dog. Most pet dogs have a name and an age, and most can sit and roll over — so those two attributes and two behaviors go into our Dog class:

class Dog: """A simple attempt to model a dog.""" def __init__(self, name, age): """Initialize name and age attributes.""" self.name = name self.age = age def sit(self): print(f"{self.name} is now sitting.") def roll_over(self): print(f"{self.name} rolled over!")
  • The class keyword starts the definition; by convention, capitalized names refer to classes. No parentheses here — we're building this class from scratch.
  • A function that's part of a class is a method. __init__() is a special method that Python runs automatically every time a new instance is created — the double underscores keep Python's default names from colliding with yours.
  • self must be the first parameter of every method: when Python calls the method later, it passes the instance itself into self automatically.
  • self.name = name takes the value of the parameter name and attaches it to the instance being created. Variables reachable through an instance like this are its attributes.

Making instances

my_dog = Dog('Willie', 6) # constructor expression — __init__ runs here print(my_dog.name) # attribute → 'Willie' (no parentheses) my_dog.sit() # method → 'Willie is now sitting.' your_dog = Dog('Lucy', 3) # a second, separate instance print(my_dog == your_dog) # False — two distinct objects in memory

The line Dog('Willie', 6) is a constructor expression: Python calls __init__() with those arguments (self is passed for you), builds an instance representing this particular dog, and returns it. You reach everything inside with dot notation. One class, many objects — Willie and Lucy each carry their own name and age, yet both can sit().

PREREQ P8.2 · WORKING WITH CLASSES AND INSTANCES

P8.2 Working with classes & instances

Once a class exists, most of your time goes into working with the instances made from it — especially modifying their attributes. Here is a Car class with a twist: one attribute gets a value without being passed in as a parameter.

class Car: """A simple attempt to represent a car.""" def __init__(self, make, model, year): self.make = make self.model = model self.year = year self.odometer_reading = 0 # default value — no parameter needed def read_odometer(self): print(f"This car has {self.odometer_reading} miles on it.") def update_odometer(self, mileage): """Set the reading — reject any attempt to roll it back.""" if mileage >= self.odometer_reading: self.odometer_reading = mileage else: print("You can't roll back an odometer!") def increment_odometer(self, miles): """Add the given amount to the odometer reading.""" self.odometer_reading += miles

You can change an attribute three ways — directly through the instance, through a method that sets it, or through a method that increments it:

my_new_car = Car('audi', 'a4', 2025) my_new_car.odometer_reading = 23 # 1. change the attribute directly… my_new_car.update_odometer(100) # 2. …or let a method guard the change my_new_car.update_odometer(50) # → You can't roll back an odometer! my_new_car.increment_odometer(10) # 3. or add to it through a method my_new_car.read_odometer() # This car has 110 miles on it.

The method version wins whenever an update needs logic: update_odometer() checks that the new reading makes sense before touching the attribute. Methods like this let you control how your class gets used.

__repr__ and __str__: how your object introduces itself

Evaluate my_new_car in a cell and Python answers with an unhelpful memory address. Two special methods fix that: __repr__() returns the "official" string representation (what repr() and a bare cell evaluation show), and __str__() returns the friendly version used by print() and str():

def __repr__(self): return f'Car(make={self.make}, model={self.model}, year={self.year})' def __str__(self): return f"Created in {self.year} {self.make} {self.model}".title() # my_new_car → Car(make=audi, model=a4, year=2025) ← __repr__ # print(my_new_car) → Created In 2025 Audi A4 ← __str__
🔍 DUNDER METHODS

Special methods like __init__(), __repr__(), and __str__() are called dunder methods (Double UNDERscore). There are many more, and they are how Python's built-in machinery — printing, comparing, even the + operator (see P8.5) — hooks into your classes.

PREREQ P8.3 · INHERITANCE

P8.3 Inheritance: building on what exists

You don't always have to start from scratch. If your new class is a specialized version of one you already have, use inheritance — the is-a relationship. The original class is the parent (base) class; the new one is the child (derived) class, and it takes on the parent's attributes and methods while adding its own. An electric car is a car, so:

class ElectricCar(Car): # ← parent class in parentheses """Represent aspects of a car, specific to electric vehicles.""" def __init__(self, make, model, year): super().__init__(make, model, year) # run Car's __init__ first… self.battery_size = 40 # …then add what's new def describe_battery(self): print(f"This car has a {self.battery_size}-kWh battery.") def fill_gas_tank(self): # ← same name as Car's method: OVERRIDE print("This car doesn't have a gas tank!")
  • The parent class must be defined before the child, and its name goes in parentheses in the child's definition.
  • super() is a special function that calls a method from the parent class — here it runs Car's __init__(), so every ElectricCar gets make, model, year, and the odometer for free. (The name comes from calling the parent a superclass and the child a subclass.)
  • To override a parent method that doesn't fit the child, define a method in the child with the same name — filling a gas tank is meaningless for an EV, so ElectricCar replaces it. Python then ignores the parent's version.
my_leaf = ElectricCar('nissan', 'leaf', 2025) print(my_leaf.get_descriptive_name()) # inherited from Car my_leaf.describe_battery() # new in ElectricCar my_leaf.fill_gas_tank() # This car doesn't have a gas tank! isinstance(my_leaf, ElectricCar), isinstance(my_leaf, Car) # (True, True)
🔍 TYPE-CHECKING AN OBJECT

type(my_leaf) tells you the exact class, but for checking "is this thing a Car?" prefer isinstance(): it returns True if the object is of the given class or any subclass of it — so an ElectricCar counts as a Car, which is usually what you mean.

Composition: the other way to combine classes

As a class grows, part of it may deserve to be a separate class. Breaking a large class into smaller classes that work together is called composition — the has-a relationship. If ElectricCar keeps accumulating battery attributes and methods, move them into a Battery class and store an instance of it as an attribute:

class Battery: """A simple attempt to model a battery for an electric car.""" def __init__(self, battery_size=40): # optional parameter, default 40 self.battery_size = battery_size def get_range(self): if self.battery_size == 40: range = 150 elif self.battery_size == 65: range = 225 print(f"This car can go about {range} miles on a full charge.") class ElectricCar(Car): def __init__(self, make, model, year): super().__init__(make, model, year) self.battery = Battery() # an ElectricCar HAS A Battery my_leaf = ElectricCar('nissan', 'leaf', 2025) my_leaf.battery.get_range() # reach through the attribute

PREREQ P8.4 · ENCAPSULATION & CLASS METHODS

P8.4 Encapsulation, class methods & class attributes

Most OOP languages let you encapsulate — hide — an object's data, making it private. Python does not have truly private data; instead it relies on naming conventions: any attribute name beginning with an underscore (_hour) is meant for the class's internal use only, and outside code should go through the class's methods instead. Names without a leading underscore are considered public. A getter method returns an internal value; a setter method validates a new value before storing it — together they form the class's public interface:

class Time: """Class Time with read-write attributes.""" def __init__(self, hour=0, minute=0, second=0): self.set_hour(hour) # even the constructor uses the setter def get_hour(self): # getter — read the value return self._hour # _hour: internal use by convention def set_hour(self, hour): # setter — validate, then write if not (0 <= hour < 24): raise ValueError(f'Hour ({hour}) must be 0-23') self._hour = hour wake_up = Time(hour=11, minute=30) wake_up.get_hour() # 11 — through the public interface wake_up.set_hour(8) # instead of wake_up._hour = 8 wake_up._hour # still works! _ is a convention, not a lock

Methods can be internal too: a utility method that only serves the class's inner workings gets a single leading underscore, the same way other languages would mark it a private method. Doubling the underscore goes one step further — Python name-mangles __x into _ClassName__x, so naive outside access fails:

class PrivateClass: def __init__(self): self.public_data = "public" # public attribute self._private_data = "private1" # internal by convention self.__private_data = "private2" # name-mangled my_object = PrivateClass() my_object.public_data # 'public' my_object._private_data # 'private1' — reachable, just frowned upon my_object.__private_data # AttributeError! my_object._PrivateClass__private_data # 'private2' — the mangled name still works
⚠️ "PRIVATE" IS A HANDSHAKE, NOT A LOCK

Even with conventions in place, attributes are always accessible: _hour can be read directly, and __private_data only looks hidden — Python simply renamed it to _PrivateClass__private_data. Encapsulation in Python is an agreement between programmers, which is exactly why library docs matter: names starting with _ are the author telling you "don't rely on this." (Python also offers a more Pythonic getter/setter mechanism called properties — worth a look once this pattern feels natural.)

Class methods and class attributes

Class methods belong to the class rather than to individual objects. You recognize one by two markers: the @classmethod decorator above its def, and cls as the first parameter — cls is to the class what self is to an instance, so a class method cannot touch any individual object's attributes. A class attribute is a variable created inside the class but outside all methods; it is shared by every instance:

class CreateCounter: count = 0 # class attribute — shared by ALL instances def __init__(self): CreateCounter.count += 1 @classmethod def how_many(cls): # cls = the class itself, not an instance return cls.count a = CreateCounter(); b = CreateCounter(); c = CreateCounter() print(CreateCounter.how_many()) # 3 — no instance needed to call it

The most frequent real use of class methods is as alternative constructors: the notebook's AsciiArt class can be built from a string with __init__() or from a file with AsciiArt.fromFile(filename) — a class method that reads the file and then returns AsciiArt(characters), reusing __init__() so the logic lives in one place (the DRY principle). There are also static methods (@staticmethod, no self or cls) — effectively plain functions parked inside a class. You'll rarely need any of these; the notebook's own advice is that if you're thinking "why can't I just use a function or a global variable?", you probably don't need them. The goal here is only to recognize them in code you read.

PREREQ P8.5 · POLYMORPHISM & DATA CLASSES

P8.5 Polymorphism & data classes

Polymorphism lets objects of one type be treated as objects of another. You've met it constantly without the name: len() happily takes a string, a list, or a dictionary (a generic function — function overloading), and + adds two numbers but concatenates two strings (operator overloading). In your own classes, the classic route is method overriding: define a method in a base class, override it in each derived class, and then treat every object through the same interface:

class Animal: def __init__(self, name): self.name = name def speak(self): pass # do nothing — each child fills this in class Dog(Animal): def speak(self): return "Woof!" class Cat(Animal): def speak(self): return "Meow!" for animal in [Dog("Rufus"), Cat("Whiskers"), Dog("Buddy")]: print(f'{animal.name} : {animal.speak()}') # Rufus : Woof! · Whiskers : Meow! · Buddy : Woof!
🎯 THE PAYOFF

One loop, one method name, many classes — the loop never asks which kind of animal it holds. Any function that calls animal.speak() works for every current and future subclass of Animal. This is why library code can say "give me anything array-like": the interface, not the exact class, is what matters.

Operator overloading with dunder methods

Dunder methods let your objects work with Python's built-in operators. Define __add__() and + works on your class; define __eq__() and == compares by value instead of by identity:

class Point: def __init__(self, x, y): self.x = x self.y = y def __add__(self, other): # runs when you write p1 + p2 return Point(self.x + other.x, self.y + other.y) def __eq__(self, other): # runs when you write p1 == p2 return (self.x == other.x) and (self.y == other.y) p3 = Point(1, 2) + Point(3, 4) print(p3.x, p3.y) # 4 6 print(p3 == Point(4, 6)) # True

This is exactly how numpy arrays make a + b mean element-wise addition — the library authors wrote the dunder methods so you can write plain arithmetic.

Data classes — one line of recognition

Python 3.7 added data classes: decorate a class with @dataclass and Python autogenerates the boilerplate — __init__(), __repr__(), __eq__(), and optionally the comparison operators — from a short list of annotated attributes. Less code to write, less to maintain and debug, and static analysis tools can use the type annotations to flag mistakes before you run. Optional here: just recognize @dataclass as "a class whose plumbing was generated automatically."
