PREREQ P7.1 · READING FROM A FILE

P7.1 Reading a file

Variables are a fine way to store data while your program is running, but if you want data to persist after the program has finished, you need to save it to a file. You can think of a file's contents as a single string value — potentially gigabytes in size. To do any work with a file, even just printing its contents, you first open it. The open() function takes the file's name, looks for it in the directory where the running program lives, and returns a file object you work with:

file_object = open('pi_digits.txt') # returns an object representing the file print(file_object.read()) # read() → the whole contents as one long string file_object.close() # you must remember to close it yourself…

The recommended way skips the bookkeeping. The keyword with closes the file once access to it is no longer needed — notice we call open() but never close():

# The recommended way: with open('pi_digits.txt') as file_object: contents = file_object.read() # one long string; no close() needed print(contents.strip()) # strip() removes the trailing blank line
📌 WHY with?

You could pair open() and close() yourself — but if a bug prevents the close() line from running, the file may never close, and data can be lost or corrupted. It is not always easy to know exactly when to close a file; with the with block, Python figures that out for you and closes the file automatically when the block finishes.

Reading line by line

Often you want to examine a file one line at a time — looking for certain information, or modifying the text in some way. The file object is iterable, so a plain for loop does it:

filename = 'pi_digits.txt' with open(filename) as file_object: # the file object is iterable! for line in file_object: print(line.strip())

Making a list of lines

The file object returned by open() is only available inside the with block. To keep access to the contents afterwards, store the lines in a list with readlines() and work with the list:

with open(filename) as file_object: lines = file_object.readlines() # a list of strings — survives the block pi_string = '' for line in lines: pi_string += line.strip() # glue the digits, dropping each newline print(pi_string) # 3.141592653589793238462643383279 print(len(pi_string)) # 32 — counts the leading 3 and the decimal point
⚠️ THREE READING TRAPS
  • Everything you read is a string. Read in a number and you get '3.14', not 3.14 — convert with int() or float() before doing math.
  • Doubled newlines. Each line of the file ends in \n, and print() adds its own — so a naive loop prints a blank line after every line. strip() each line to fix it.
  • read() at the end of the file returns an empty string — it does not raise an error. That empty string is what shows up as the extra blank line after read(), and why a second read() in a row prints nothing.
PREREQ P7.2 · FILE PATHS

P7.2 File paths

Sometimes the file you want won't be in the same directory as your program. To open a file from somewhere else, you give open() a file path — directions to a specific location on your system. A relative file path starts from the directory where the currently running program lives:

with open('text_files/pi_digits2.txt') as file_object: # look inside the folder text_files, contents = file_object.read() # assumed to sit in the current directory

An absolute file path instead spells out exactly where the file is on the computer, regardless of where the program runs from. Absolute paths tend to be long, so it's helpful to assign them to a variable and pass that to open():

# Linux / macOS — slashes, starting from the root file_path = '/home/Users/Desktop/text_files/pi_digits2.txt' with open(file_path) as file_object: contents = file_object.read() # Windows — drive letter and backslashes (note the r'…' raw string) file_path = r'C:\Users\adm\Desktop\text_files\pi_digits2.txt'
🔍 SLASH OR BACKSLASH?

Windows displays paths with backslashes (\), but you can still use forward slashes (/) in your Python code — they work on every system. If you do write backslashes, prefix the string with r (a raw string) so Python doesn't mistake \t or \n for escape characters.

PREREQ P7.3 · WRITING & APPENDING

P7.3 Writing & appending

One of the simplest ways to save data is to write it to a file: the output is still there after your program ends, you can examine it later, share it, or write another program that reads it back in. To write, call open() with a second argument — the mode:

filename = 'programming.txt' with open(filename, 'w') as file_object: # 'w' = write mode — overwrites the contents! file_object.write("We love programming!")
ModeMeaningIf the file existsIf it doesn't
'r' (default)read only — what you get when you omit the modereads itFileNotFoundError
'w'writeerases it firstcreates it
'a'append — new text goes at the endkeeps itcreates it
'r+'read and writereads/writesmust already exist
⚠️ THREE WRITING TRAPS
  • 'w' erases before it writes. open() creates the file if it doesn't exist — but if it does exist, write mode wipes its contents before returning the file object.
  • Only strings go in. file_object.write(12) fails — Python can only write strings to a text file, so convert numbers with str() first.
  • No free newlines. write() doesn't add \n for you — include the newline characters yourself. (There is also writelines() for writing a list of strings.)

Appending to a file

To add content without writing over what's already there, open the file in append mode: nothing is erased, every line you write lands at the end, and a missing file is created empty for you.

with open(filename, 'a') as file_object: # 'a' = append mode file_object.write("\nWe also love finding meaning in large datasets.\n") file_object.write("We love creating apps that can run in a browser.\n")

The file-position pointer

While reading through a file, the system maintains a file-position pointer — an index marking where the next character will be read or written. tell() reports the current position and seek() moves it. That is why this 'r+' snippet ends up appending: reading everything first pushes the pointer to the end, so the write lands there.

year = 2025 with open(filename, 'r+') as file_object: # the file must already exist! spam = file_object.readlines() # reading moves the pointer to the end… print(file_object.tell()) # …tell() reports that position file_object.write(str(year)) # …so this write appends
PREREQ P7.4 · STORING DATA WITH JSON

P7.4 Storing data with json

Many programs collect information from users — preferences in a game, data for a visualization — and store it in data structures like lists and dictionaries. When the user closes the program, you almost always want to save what they entered. Writing a list to a text file by hand is clumsy; the json module lets you dump a simple Python data structure into a file and load it back the next time the program runs.

import json numbers = [2, 3, 5, 7, 11, 13] filename = 'numbers.json' with open(filename, 'w') as f: json.dump(numbers, f) # two arguments: the data, then the file object

A second program — or the same one, next week — reads the list back into memory with json.load():

with open(filename) as f: numbers = json.load(f) print(numbers) # [2, 3, 5, 7, 11, 13] — a real list again, not a string
🎯 WHY JSON?

Two reasons to reach for json: it saves whole data structures (lists, dictionaries) so your users never have to start over each time they run the program — and it is a portable, language-independent format, so the same file can be shared with programs written in other languages.

PREREQ P7.5 · EXCEPTIONS

P7.5 Exceptions: try / except

Python manages errors that arise while a program runs with special objects called exceptions. Whenever an error makes Python unsure what to do next, it creates an exception object. If you write code that handles the exception, the program keeps running; if you don't, the program halts and shows a traceback — a report of the exception that was raised:

print(5/0) # Traceback (most recent call last): … # ZeroDivisionError: division by zero ← the exception object, named on the last line

When you suspect an error may occur, wrap the risky line in a try-except block: Python tries the code, and if it raises the named exception, runs the except block instead of crashing. The user sees a friendly message you wrote, not a traceback:

try: print(10/0) except ZeroDivisionError: print("You can't divide by zero!", end=" ") finally: print("terminated") # → You can't divide by zero! terminated

If the code in the try block works, Python skips the except block entirely. And if you write except: with no exception type, it captures all exceptions — convenient, but it can also hide errors you didn't anticipate.

Using exceptions to prevent crashes

Handling errors matters most when the program has more work to do after the error occurs. This division-only calculator keeps running no matter what the user types, thanks to two more clauses — else and finally:

print("Give me two numbers, and I'll divide them.") print("Enter 'q' to quit.") while True: first_number = input("\nFirst number: ") if first_number == 'q': break second_number = input("Second number: ") if second_number == 'q': break try: answer = int(first_number) / int(second_number) except ZeroDivisionError: print("You can't divide by 0!") else: # runs only if the try block succeeded print(answer) finally: # always runs — success or exception print("\nGive me two numbers, and I'll divide them.")
ClauseWhen it runsWhat belongs there
tryalways attempted firstonly the code that might raise the error
exceptonly if the named exception is raised in trythe friendly message / recovery code
elseonly if the try block succeededcode that depends on try having worked
finallyguaranteed — success or exceptioncleanup that must happen either way

Handling the FileNotFoundError exception

One common issue when working with files is a missing file: it's in a different location, the name is misspelled, or it doesn't exist at all. Opening it in read mode then raises a FileNotFoundError — exactly the kind of exception a try-except block turns into a friendly message:

filename = 'alice.txt' with open(filename) as f: # read mode — the file must exist contents = f.read() # FileNotFoundError: [Errno 2] No such file or directory: 'alice.txt'
