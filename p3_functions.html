PREREQ P3.1 · WHY FUNCTIONS?

P3.1 Why functions? def, calls & parameters

The best way to develop and maintain a large program is to build it from smaller pieces — a technique called divide and conquer. In programming, abstraction means hiding the complexity of an algorithm's sub-steps inside a function: once the function is built, you can treat it as a simple expression with defined inputs and outputs, and use it without knowing its internal details.

You have already been using functions — print(), str(), and len() are Python's built-in ones, parentheses wrapped around their arguments. A function gives a name to a series of operations; writing that name in an expression or statement is a function call. Two payoffs make this worth learning well:

  • You can execute a block of code from various places in your program by calling it — instead of duplicating the code.
  • Programs become easier to modify: change a function's code once, and every call runs the updated version.

def statements with parameters

def hello(name): print('Hello,', name) hello('Alice') # → Hello, Alice hello('Bob') # → Hello, Bob print(name) # NameError! the parameter is gone once the call ends

The def statement defines the hello() function; the indented lines below it make up the function's body. The line hello('Alice') calls the function — also described as passing the string value 'Alice' to it. Here name is a parameter: a variable in the definition that receives the argument's value when the function is called. Execution enters the function, name is automatically set to 'Alice', and the body prints the greeting.

One thing to note: the value stored in a parameter is forgotten when the function returns. That is why the final print(name) above raises a NameError — after hello('Bob') finishes, no variable named name exists any more.

⚠️ TWO BEGINNER MISTAKES
  • Forgetting the pair of parentheses after the function name — especially easy when the function takes no parameters: it's hello(), not hello.
  • Overlooking the indentation that separates the function's body from the code that calls the function — the call must be back at the left margin, not inside the body.
📌 A FUNCTION IS A BLACK BOX

Think of a function as a black box: input in the form of parameters, output in the form of a return value, and the code inside doesn't affect variables in other functions. This limits what code could be changing your variables — which is exactly what makes functions such a good debugging tool.

PREREQ P3.2 · RETURN VALUES & NONE

P3.2 Return values & the None value

When you call len('Hello'), the call evaluates to the integer 5. The value a function call evaluates to is its return value. In your own functions you specify it with a return statement: the return keyword, followed by the value or expression the function should hand back. Functions that return values are sometimes called fruitful functions.

import random def getAnswer(answerNumber): if answerNumber == 1: return 'It is certain' elif answerNumber == 2: return 'It is decidedly so' elif answerNumber == 3: return 'Yes' r = random.randint(1, 3) fortune = getAnswer(r) # the returned string lands in fortune print(fortune)

Trace the execution: Python imports random, then defines getAnswer() — defining is not calling, so execution skips over the body. random.randint(1, 3) picks a number, and calling getAnswer(r) moves execution to the top of the function with r stored in the parameter answerNumber. The matching return sends a string back to the line that made the call, where it is assigned to fortune and printed.

🔍 RETURN EXITS IMMEDIATELY

The moment execution reaches a return statement, the function stops — any statements written after it never run. A print() placed below a return is unreachable code: a very common slip, and one of the quiz questions below is waiting to catch you on it.

The None value

Python has a value called None that represents the absence of a value — it is the only value of the NoneType data type, useful when you need something that won't be confused for a real value. Every function call must evaluate to something, so a function with no return statement returns None: behind the scenes Python adds return None to the end of its body, and a bare return with no value does the same. A function that returns no value is called a void function — print() is one: it displays text on the screen, but that is a side effect, not a return value.

spam = print('Hello!') # prints Hello! … print(spam) # … but the call evaluated to None type(spam) # <class 'NoneType'>
⚠️ PRINT IS NOT RETURN
def add_em(x, y, z): print(x + y + z) # displays the sum — but returns nothing a = add_em(1, 2, 3) # prints 6 print(a) # None

Writing print where you mean return is a classic trap, and it is hard to spot because the output looks identical when you run the function by itself. The difference appears the moment you try to use the result: a printed value is gone; a returned value can be stored, passed on, and computed with.

PREREQ P3.3 · KEYWORD & DEFAULT ARGUMENTS

P3.3 Positional, keyword & default arguments

When you call a function, Python must match each argument in the call with a parameter in the definition. The simplest matching is by order — values matched this way are positional arguments:

def describe_pet(animal_type, pet_name): """ Display information about a pet. we can write multiple lines here! """ print("\nI have a", animal_type + ".") print("My", animal_type + "'s name is", pet_name.title() + ".") describe_pet('Pokemon', 'Harry') # order decides: animal_type='Pokemon', pet_name='Harry'

The triple-quoted string at the top of the body is a docstring — a multi-line comment describing what the function does. Python looks for it right after the definition when generating documentation, and help(describe_pet) prints it together with the function's name and parameters. Try help(print) too: you'll find that print() itself accepts a keyword argument — sep replaces the default single-space separator between multiple values:

print("8", "9", sep="*") # → 8*9

Keyword arguments

A keyword argument is a name-value pair you pass to the function. Because you directly associate each value with a parameter name, there is no confusion about roles, and you no longer have to worry about ordering the arguments correctly:

describe_pet(animal_type='Pokemon', pet_name='Harry') # same call, names decide

Default parameter values

A default parameter has a pre-assigned value: if the call provides an argument, Python uses it; if not, the default steps in. If most of your calls describe dogs, make 'dog' the default:

def describe_pet(pet_name, animal_type='dog'): ... describe_pet('willie') # → I have a dog. My dog's name is Willie. def describe_pet(animal_type='dog', pet_name): # SyntaxError! ...

Notice the parameter order had to change: 'willie' is still interpreted as a positional argument, so it must match the first parameter. The rule: any parameter with a default value must be listed after all the parameters without defaults — that is what lets Python keep interpreting positional arguments correctly, and violating it is a SyntaxError.

Arbitrary numbers of arguments: *args and **kwargs (optional)

Sometimes you can't know ahead of time how many arguments a call will provide. A parameter written *toppings collects all remaining positional arguments into a tuple (P4 territory); one written **user_info collects extra name-value pairs into a dictionary (P5 territory). You'll often see the generic names *args and **kwargs — and **kwargs must be the rightmost parameter.

def make_pizza(size, *toppings): """Print the list of toppings that have been requested.""" print("The size of the pizza is", size, "inch with the following toppings:") print(toppings) make_pizza(6, 'pepperoni') make_pizza(8, 'mushrooms', 'green peppers', 'extra cheese') def build_profile(first, last, **user_info): """Build a dictionary containing everything we know about a user.""" user_info['first_name'] = first user_info['last_name'] = last return user_info user_profile = build_profile('albert', 'einstein', location='princeton', field='physics')

PREREQ P3.4 · LOCAL VS GLOBAL SCOPE

P3.4 Local vs global scope

Parameters and variables assigned inside a called function exist in that function's local scope; variables assigned outside all functions exist in the global scope. A variable is a local variable or a global variable — it must be one or the other, never both. Think of a scope as a container for variables: when the scope is destroyed, everything in it is forgotten.

📌 SCOPE LIFETIMES

There is one global scope per module, created when your program begins. A local scope is created every time a function is called and destroyed when the function returns — so the next call starts fresh: local variables never remember values from the previous call.

Local variables cannot be used in the global scope

def spam(): eggs = 31337 spam() print(eggs) # NameError — eggs lived only in spam()'s local scope

The error happens because eggs exists only in the local scope created when spam() is called. Once execution returns, that scope is destroyed — there is no longer any variable named eggs.

Each call gets its own local scope

A new local scope is created whenever a function is called — including when a function calls another function, so multiple local scopes can exist at the same time, and one function's locals are invisible to another's. Meanwhile, a global variable can be read from inside a function, as long as nothing in the function assigns to that name:

def spam(): eggs = 'spam local' print(eggs) # prints 'spam local' def bacon(): eggs = 'bacon local' print(eggs) # prints 'bacon local' spam() print(eggs) # prints 'bacon local' — spam() never touched bacon()'s eggs eggs = 'global' bacon() print(eggs) # prints 'global'

Three different variables, one name: bacon()'s eggs, spam()'s eggs, and the global eggs all coexist without interfering. If you want a function to modify the global variable rather than create its own local one, declare it with the global keyword:

def spam(): global eggs # now assignments target the global eggs eggs = 'spam local' eggs = 'global' spam() print(eggs) # → 'spam local' — the global was modified
⚠️ GO EASY ON GLOBAL VARIABLES

Don't use global variables unless there is a very good reason. They invite unintentional changes that propagate through the entire program: a change that makes sense for one function may quietly break the processing in every other function that shares the variable.

PREREQ P3.5 · MODULES

P3.5 Storing your functions in modules

Functions separate blocks of code from your main program, and descriptive names make programs easier to follow. You can go a step further and store your functions in a separate file — a module is simply a file ending in .py whose code you can bring into your program with an import statement. Storing functions this way lets you hide implementation details and focus on higher-level logic, reuse the same functions in many programs, share your file with other programmers without sharing the whole program — and use the function libraries other people have written.

# pizza.py — a module is just a .py file with code in it def make_pizza(size, toppings): """Summarize the pizza we are about to make.""" print("\nMaking a " + str(size) + "-inch pizza with the following toppings:") print(toppings)

Now any program in the same folder can import it — four flavors, from most explicit to most abbreviated:

import pizza # whole module → call with dot notation pizza.make_pizza(16, 'pepperoni') from pizza import make_pizza # one function → call by bare name make_pizza(12, 'mushrooms') from pizza import make_pizza as mp # function alias mp(16, 'pepperoni') import pizza as p # module alias p.make_pizza(12, 'mushrooms')

When Python reads import pizza, it opens pizza.py and makes its functions available — the copying happens behind the scenes just before the program runs. The as alias forms are handy when a function's name is long or would conflict with an existing name in your program. There is also from pizza import *, which copies every function into your file so you can call each by bare name — but it's best avoided with larger modules you didn't write: if the module defines a name that matches one already in your project, you can get unexpected results.

🎯 THE BIG PICTURE

Functions are the primary way to organize your code into logical groups. Because their variables live in local scopes, the code in one function cannot directly affect the values of variables in other functions — inputs come in as parameters, outputs go out as return values, and everything in between stays inside the box. That single idea is what makes big programs debuggable.
