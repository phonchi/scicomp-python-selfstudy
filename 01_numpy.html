Explain why NumPy's ndarray — not the built-in list — is the main data structure of scientific computing, and install / import NumPy in your own environment.

Array-oriented programming means expressing computations on whole arrays at once — 1.0 / values instead of a for loop over every element. Two big reasons to prefer it:

Built-in listNumPy array
Element typesHeterogeneous (anything)Homogeneous (one dtype)
Memory layoutBoxes + pointers (large)One compact block
LoopsExplicit, interpreted, slowInternal iteration in C — fast
TypingDynamically typed elementsStatically typed elements (dtype fixed at creation)

Because every element of an array shares one type (its dtype), NumPy stores the raw numbers in a single contiguous block — less memory, and the loop happens inside compiled code instead of the Python interpreter.

⚠ TWO CLASSIC TRAPS

np.arange(a, b) excludes the endpoint b, but np.linspace(a, b, n) includes it by default. And np.random.randint(1, 7) excludes 7 — unlike the built-in random.randint(1, 6), which includes 6!

When you want «N evenly spaced numbers over an interval», linspace() is the tool — this exact phrasing shows up in the quiz.

Slice = view!

Unlike list slicing (which copies), array slicing returns a view: writing through the slice changes the original. Use .copy() when you need independence.

View or copy? The rule of thumb

OperationReturnsChanging it affects the original?
Slicing a[0:3]viewYes ⚠
reshape() / ravel() / .Tview*Yes ⚠ (*when possible)
flatten()copyNo
Fancy indexing a[[1, 3]]copyNo
.copy()copy (deep)No

🎯 AFTER THIS VIDEO YOU CAN…

Replace explicit loops with vectorized operators (+ − * / ** +=) that act element-wise on whole arrays — and explain where the ~100× speedup comes from.

The video's running example — take the reciprocal of a million numbers:

def compute_reciprocals(values): # the loop version output = np.empty(len(values)) for i in range(len(values)): output[i] = 1.0 / values[i] # type-check + dispatch on EVERY element return output %%timeit compute_reciprocals(big_array) # ≈ seconds %%timeit (1.0 / big_array) # ≈ milliseconds — same math, ~100× faster

Why so slow? Each pass through a Python loop re-does type checking and function dispatch. The vectorized version pushes the loop into compiled C code where the dtype is checked once.

Decide — before NumPy tells you — whether two shapes are compatible, and use broadcasting to avoid manual copying or reshaping.

The rule, in three steps

📐 BROADCASTING RULES
  1. Pad on the left: if the arrays differ in number of dimensions, the shorter shape is padded with 1s on its left side.
  2. Stretch the 1s: any dimension of size 1 is stretched (virtually — no memory copied) to match the other array's size in that dimension.
  3. Otherwise fail: if the sizes in a dimension differ and neither is 1 → ValueError: operands could not be broadcast together.

Explain why ufuncs are fast (compiled routines behind a Python interface), use them with broadcasting, and vectorize your own scalar function with np.vectorize().

Every operator you met in 1.5 is secretly a universal function: a + b calls np.add(a, b), a * 5 calls np.multiply(a, 5). Ufuncs operate element-wise, obey broadcasting, and run as compiled C routines — the Python layer is just the interface.

CategoryUfuncs
Arithmeticnp.add, np.subtract, np.multiply, np.divide, np.power
Mathnp.sqrt, np.exp, np.log, np.abs
Trigonometrynp.sin, np.cos, np.tan
Comparisonnp.greater, np.less, np.equal, …

Custom scalar logic (like a Heaviside step function with a threshold) can't use operators directly — wrap it:

def Theta(x, th): # scalar version: if/else on one number return 1 if x >= th else 0 Theta_vec = np.vectorize(Theta) # now it maps over whole arrays Theta_vec(np.array([-3, -1, 0, 2]), 1) # [0 0 0 1]
⚠ HONEST FINE PRINT

np.vectorize() gives you the convenience of array syntax, but under the hood it still calls your Python function per element — it is not compiled-C fast. True speed needs real ufuncs or rewriting with array operations.

🏁 PERFORMANCE CHECKLIST (MODULE SUMMARY)
  • Prefer vectorized operations / ufuncs over explicit loops.
  • Use views (slicing, reshape) instead of copies when you don't need independence.
  • Use augmented assignment (a += b) to update in place without allocating a new array.
  • Choose the smallest dtype that safely holds your data.
  • Store large arrays in binary format.
