Explain when Pandas' labeled data structures beat plain NumPy arrays — and when they don't — and install / import Pandas in your own environment.

NumPy arrays are perfect for clean, homogeneous grids of numbers. But most real data looks like a spreadsheet or a database table: rows have names, columns hold different types, and values are sometimes simply missing. Pandas — built on top of NumPy — adds exactly those three things, which is why it is the standard tool for data munging (cleaning and reshaping raw data until it is ready for analysis).

NumPy ndarrayPandas Series / DataFrame
Row / column labelsImplicit integers onlyExplicit Index — strings, dates, anything

Create a Series from a list, an array, or a dictionary — and explain why loc / iloc beat plain square-bracket indexing.

A Series is a one-dimensional labeled array — one column of data. Two mental models, both useful:

  • A generalized NumPy array — the index is explicit and can be any type (strings, dates…), not just positions 0, 1, 2….
  • A specialized dictionary — keys map to values, but the values are typed and stored contiguously, so vectorized operations still fly.
population_dict = {'New Taipei City': 4_013_659, 'Taichung City': 2_826_230, 'Kaohsiung City': 2_733_964, 'Taipei City': 2_465_610, …} population = pd.Series(population_dict) # index = the dict keys population['New Taipei City'] # 4013659 — dictionary-style access population['New Taipei City':'Kaohsiung City'] # label slice — endpoint INCLUDED!
⚠ THE INTEGER-INDEX TRAP

When the explicit index itself contains integers, plain brackets become ambiguous: data[1] uses the explicit label 1, but data[1:3] slices by implicit position! "Explicit is better than implicit" — so Pandas gives you two indexer attributes that never guess:

data.loc[…] always uses the explicit labels · data.iloc[…] always uses the implicit positions.

ExpressionWhich index?data = Series(['a','b','c'], index=[1,3,5])
data[1]explicit label'a'
data[1:3]implicit position ⚠'b', 'c'
data.loc[1]always explicit'a'
data.iloc[1]always implicit'b'

Build a DataFrame three different ways, and read its anatomy — index (row labels), columns (column labels), values (the underlying NumPy data).

A DataFrame is the 2-D sibling of the Series: think of it either as a dictionary of Series that share one index (each column is a Series), or as a generalized 2-D array with explicit row and column labels. Unlike a NumPy matrix, each column may hold a different dtype — exactly like a spreadsheet or a database table.

You haveConstructor callNote
A dict of Series (or lists)pd.DataFrame({'area': area, 'subdiv_cnt': subdiv})keys → column names; indexes are aligned
A list of dictspd.DataFrame([{'a':1,'b':2}, {'b':3,'c':4}])missing keys are filled with NaN
A 2-D NumPy arraypd.DataFrame(arr, index=[…], columns=[…])labels supplied separately
area = pd.Series({'New Taipei City': 2052.57, 'Taichung City': 2214.90, …}) subdiv = pd.Series({'New Taipei City': 29, 'Taichung City': 29, …}) cities = pd.DataFrame({'area': area, 'subdiv_cnt': subdiv}) # dict of Series cities.index # Index(['New Taipei City', 'Taichung City', …]) — row labels cities.columns # Index(['area', 'subdiv_cnt']) — column labels (also a pd.Index!) cities.values # the underlying 2-D NumPy array (or cities.to_numpy())

Selection cheat table
cities['area']column → Series
cities.loc[label]row by label

loc includes, iloc excludes

loc slices by explicit label and includes the endpoint; iloc slices by implicit position with Python semantics — endpoint excluded. Direct indexing (cities['area']) means columns; slicing (cities[1:4]) means rows.

Adding and removing data

cities['area_per_div'] = cities['area'] / cities['subdiv_cnt'] # new column, Series-style math cities.drop('Taoyuan City') # drop a ROW — returns a modified COPY cities.drop('subdiv_cnt', axis=1) # drop a COLUMN (axis='columns' also works) cities.drop('Taoyuan City', inplace=True) # in-place: modifies cities, returns None

Remember the in-place rule: most methods return a modified copy and leave the original untouched; pass inplace=True (or assign the result) when you really mean to change the DataFrame.

Everything you know about NumPy ufuncs still works — element-wise, vectorized, broadcasting — but Pandas adds one superpower on top: operations are label-aware.

cities.query('area > 2200') # the mask cities[cities['area'] > 2200] as a readable string cities.head(2) # first rows (default n=5) cities.tail(2) # last rows (default n=5) np.exp(series) # unary ufunc → index PRESERVED, values transformed population / area # binary ufunc → indexes ALIGNED on the union, NaN where missing A.add(B, fill_value=0) # same as A + B but missing slots count as 0

Why alignment matters

You never have to worry about row order again — data lines up by label, not by position. Misaligned or missing entries surface honestly as NaN instead of silently pairing the wrong rows.
