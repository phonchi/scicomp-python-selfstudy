SciPy (Scientific Python) is the central package for scientific operations in Python, designed to work directly on NumPy arrays. Before you write a new numerical routine, check whether SciPy already ships it: home-grown re-implementations tend to be error-prone, suboptimal, hard to share and unmaintainable, while SciPy's routines are optimized and tested by a huge community.

SciPy is organized into task-specific sub-modules that all depend on NumPy but are mostly independent of each other. This course visits four of them:

Sub-moduleFunctionalityThis module's stars
scipy.linalgLinear algebra routinessolve, det, eig, eigh
scipy.integrateIntegration routinesquad, simpson
scipy.optimizeOptimizationcurve_fit, minimize, root
scipy.statsStatisticsnorm, ttest_ind

The standard way to import them — one sub-module at a time:

from scipy import linalg # linear algebra from scipy import integrate # integration from scipy import optimize # optimization from scipy import stats # statistics
⚠ NAMESPACE FINE PRINT

The primary scipy namespace mostly contains functions that actually belong to NumPy — they are exposed for historical reasons only. That is why you practically never write import scipy: you import the sub-module you need instead.

Why scipy.linalg, not numpy.linalg?

numpy.linalgscipy.linalg
Function coverageBasic setSuperset — everything in numpy.linalg plus more advanced routines
BLAS / LAPACK backendOptional (depends on your NumPy build)Always compiled in — consistently optimized
When to prefer itOnly to avoid the SciPy dependencyScientific work — the course default

BLAS (Basic Linear Algebra Subprograms) standardizes the elementary vector/matrix operations; LAPACK builds on BLAS to solve linear systems, eigenvalue problems and SVDs (singular value decompositions — a matrix factorization you will meet later) efficiently. scipy.linalg is always backed by both.

Two ways to solve Ax = b

The video's system — 3x + 2y = 12 and 2x − y = 1 — becomes a matrix product:

A = np.array([[3, 2], [2, -1]]) b = np.array([12, 1]).reshape((2, 1)) # a column vector x = linalg.solve(A, b) # [[2.], [3.]] → x = 2, y = 3 A @ x # [[12.], [1.]] — check: reproduces b A_inv = linalg.inv(A) # route 2: the matrix inverse… x = A_inv @ b # …same answer, but see the warning below linalg.det(A) # -7.0 — nonzero ⇒ A is invertible, unique solution
⚠ SOLVE() BEATS INV() — EXAMPLE 1

Computing A⁻¹ and then multiplying is slower and less numerically stable than calling solve() directly — the course's Example 1 benchmarks both routes on a large random system. Invert a matrix only when you genuinely need the inverse itself.

Extract eigenvalues and eigenvectors of a square matrix with eig(), interpret the returned tuple, and obtain purely real eigenvalues from real symmetric matrices.

A non-zero vector v is an eigenvector of A with eigenvalue λ if A v = λ v. Rearranged, v solves the homogeneous system (A − λI) v = 0 (homogeneous = the right-hand side is all zeros) — and a non-trivial solution (v ≠ 0) can only exist when A − λI is singular, i.e. det(A − λI) = 0. The eigenvalues are the roots of that characteristic polynomial.

A = np.array([[1, 0], [0, -2]]) eigvals, eigvecs = linalg.eig(A) # tuple: 1-D eigenvalues, 2-D eigenvectors (in COLUMNS) eigvals # [ 1.+0.j -2.+0.j] — complex dtype, even for a real matrix! eigvals = eigvals.real # [ 1. -2.] — drop the zero imaginary part yourself v1 = eigvecs[:, 1].reshape(2, 1) # SECOND column ↔ second eigenvalue A @ v1 == eigvals[1] * v1 # [[True], [True]] here — but in general compare floats with np.allclose()

Symmetric matrices — where eigenvalues turn real

For a real symmetric matrix the eigenvalues are always real and the eigenvectors are mutually orthogonal. The video builds one from any random P via S = P @ P.T, then verifies v1 @ v2 ≈ 0. When you know the matrix is symmetric (or Hermitian), tell SciPy:

eig()eigh()
Inputany square matrixsymmetric / Hermitian only
Eigenvalue dtypecomplex128 — alwaysreal (float64), sorted ascending
Speed & stabilitygeneral algorithmexploits symmetry — faster, more accurate

Approximate definite integrals numerically — quad() when you can call the function, simpson() when you only have sampled points — and read the (value, error) tuple that quad() returns.

A definite integral is the limit of a sum of rectangle areas: partition [a, b] into N subintervals and add up f(xᵢ*)·Δx. Numerical integration stops short of the limit — it evaluates cleverly chosen finite sums, which is exactly what you need when no closed-form antiderivative exists (like the Gaussian e^(−x²)).

🧪 Interactive: Riemann sums — the idea inside every numerical integrator

Below is f(x) = x² + 1 on [0, 2]. The exact area is 2³/3 + 2 = 14/3 ≈ 4.66667. Drag the slider to add left-endpoint rectangles and watch the error melt away:

🧪 SANDBOX
n = 8

Left-endpoint rectangles are the crudest scheme — scipy.integrate.quad() pursues the same idea with adaptive, high-order rules and reaches ~15 digits in a handful of evaluations. And when you can't call the function at all — you only have sampled points — simpson() fits parabolas through your samples:

f = lambda x: x**2 + 1 integrate.quad(f, 0, 2) # (4.666666666666667, 5.18e-14) — (value, error estimate) xi = np.linspace(0, 2, 100) # suppose only these samples exist integrate.simpson(f(xi), x=xi) # 4.666666666666666 — from points alone g = lambda x: 2/np.pi**0.5 * np.exp(-x**2) # no antiderivative in closed form… integrate.quad(g, 0, 1.96) # …no problem for numerical integration

Least-squares curve fitting in one call — fit a·sin(b·x) to a noisy sine wave:

def test_func(x, a, b): return a * np.sin(b * x) params, params_covariance = optimize.curve_fit(test_func, x_data, y_data, p0=[2, 2]) params # ≈ [2.9, 1.5] — the amplitude and frequency that minimize Σ(residual)²

Escaping the trap

minimize(f, x0=1, bounds=((0, 10),)) confines the search to an interval; running several restarts from different x0 values and keeping the best result is the simplest global strategy.

Root finding follows the same start-point logic — and note there is no maximize(): to find a maximum you minimize −f(x) instead.

f = lambda x: x**2 + 10*np.sin(x) optimize.minimize(f, x0=0).x # [-1.306…] — global minimum optimize.minimize(f, x0=3).x # [ 3.837…] — trapped in the local minimum! optimize.root(f, x0=1).x # [ 0.] — one root… optimize.root(f, x0=-2).x # [-2.479…] — …a different guess finds the other

Use distribution objects like stats.norm (compare a sample histogram against the true pdf) and run a two-sample t-test with ttest_ind(), reading the statistic and the p-value.

scipy.stats packages statistical tools and probabilistic representations of random variables. The star is norm — not a matrix norm, but a normal (Gaussian) continuous random variable object with methods for its pdf, cdf, random sampling and more:

samples = np.random.normal(size=1000) # observations of a random variable bins = np.linspace(-4, 4, 30) bins = 0.5*(bins[1:] + bins[:-1]) # bin centers pdf = stats.norm.pdf(bins) # the TRUE density at those points plt.hist(samples, density=True); plt.plot(bins, pdf) # histogram ≈ pdf ✓

Hypothesis testing in one call

Given two sets of observations assumed to come from normally distributed populations, the t-test decides whether their means differ significantly:

samples1 = np.random.normal(0, size=50) samples2 = np.random.normal(1, size=50) # shifted mean stats.ttest_ind(samples1, samples2) # (statistic=-4.13…, pvalue=7.6e-05)
OutputMeaning
statistic (t)sign ∝ difference between the two sample means; magnitude ∝ its significance
pvalue (p)IF the two means were truly equal, how likely is data at least this extreme? Small p (< 0.05) → the equal-means assumption looks implausible → reject it; large p → the data cannot tell the means apart
